问题 / The Problem

跨链收款,又难又险

链碎片化

商户想收 Base 上的 USDC,付款人的资产却分散在 Arbitrum、Optimism、Polygon。手动跨链门槛高、易出错。

金额不确定

桥接存在滑点与费用,付款人不知道要付多少,商户也无法保证"实收"金额——结算对不上账。

条款可被篡改

收款地址、链、代币、金额都可能被恶意路由或输入改写。缺乏强校验,资金可能流向错误目标。

ChainCashier 用一个 AI Agent 把"建单 → 跨链 → 到账 → 举证"一次闭环。

方案 / The Solution

一句话建单,任意链收款

商户在对话里说出需求,GLM-5.1 解析生成"锁定收款单",应用发出一个独立支付链接;付款人打开链接、选来源链,LI.FI 算出路线——商户实收金额按预期稳定到账。

①

对话建单

GLM-5.1 结构化解析,生成锁定条款的发票。

②

跨链路由

LI.FI quote/toAmount 反算付款额,保证到账稳定。

③

链上收据

轮询状态 + 到账核验,生成不可篡改收据。

架构 / Two-Party Split

商户与付款人,彻底解耦

商户流程 · Merchant

/chat

在聊天中用自然语言描述收款需求。GLM-5.1 生成结构化发票并锁定收款条款(地址/链/代币/金额),产出独立支付链接 /pay/[invoiceId]。

付款流程 · Payer

/pay/[invoiceId]

付款人在独立浏览器会话打开链接,选择来源链。应用取 LI.FI 报价,付款人钱包签名执行路由,应用轮询状态直到铸造收据。

👛 钱包隔离 · per-role Wagmi config 🔗 发票是唯一交接物 🏪 商户钱包无需保持在线

核心功能 / Key Features

六大能力

自然语言建单

GLM-5.1 把口语描述转成结构化发票;解析失败自动回退到确定性逻辑。

跨链到账

LI.FI 在 40+ EVM 链间路由,toAmount 反算锁定商户目标到账金额。

锁定收款条款

目标地址/链/代币不匹配时 summarizePaymentQuote 直接抛错,绝不放松。

两方钱包隔离

商户与付款人各用独立 Wagmi 配置与 localStorage,演示安全互不串扰。

运行时间线

"Payment Flow Recorder"实时镜像长程任务每一步,SSE 流式逐步呈现。

链上对账收据

到账核验 + receipt/support package 哈希,全流程防篡改、可导出。

安全边界 / Safety Invariants

资金安全,绝不妥协

永不签名 · 应用只准备交易请求,交由用户钱包执行,从不托管私钥、不自动动款。

锁定条款强校验 · summarizePaymentQuote 对目标地址/链/代币不匹配抛错。

LLM 仅用于建单 · 付款路径与价值转移逻辑全为确定性,无任何 LLM 调用。

结算链受限 · 收款仅限 Base / Arbitrum;来源链由路由白名单约束。

钱包确认为唯一动款闸门 · 每一笔跨链都需用户明确授权。

收据可证 · 到账与锁定项核验,生成哈希绑定证据。
